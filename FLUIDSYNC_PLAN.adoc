= DlangUI FluidSync & Threading Implementation Plan
:toc: macro
:toc-title: Table of Contents

This document outlines the architectural changes necessary within the `dlangui` framework. It traces the emergent reasoning that led to the `fluidsync-daemon` concept and outlines the roadmap for pushing rendering outside of the strict single-threaded paradigm.

toc::[]

== The Discovery & Emergent Reasoning

This architectural plan was born from a specific observation: **When resizing a complex `dlangui` window, the user interface "ghosts" or pulls to the right, failing to lock instantly to the cursor.**

Tracking the root of this issue led to a series of cascading architectural realizations:

1. **The Ghosting Source:** The OS (Windows DWM) stretches the old window backbuffer to fill the new resize area while it waits for `dlangui` to render a new frame. 
2. **The "Sizing Trick" Mitigation:** The standard Win32 fix is to intercept `WM_SIZING` and force a synchronous redraw before the OS can stretch the pixels.
3. **The Stuttering Consequence:** Forcing a synchronous redraw in a single-threaded framework works well for simple windows. However, if the UI is complex (e.g., thousands of table rows or nested elements), the *layout math* takes longer than the mouse poll rate. This results in the mouse cursor feeling laggy or "heavy."
4. **The Need for Two-Pass "Skeleton" Rendering:** To fix the stutter while keeping the synchronous resize, we realized the rendering engine must decouple "Structural Constraints" from "Detail Rasterization". We needed a fast "skeleton" pass for immediate resizing feedback, and a deferred pass for complex details.
5. **The Budgeting Problem:** How does the single-threaded UI know when to abort detailed rendering and fall back to the skeleton? It needs a time budget strictly dictated by the monitor's 1% low frame times.
6. **The Daemon Solution (`fluidsync-daemon`):** Measuring OS ETW telemetry, DXGI swap chains, and calculating 1% lows from *within* the UI thread’s render loop incurs too much overhead via system calls. Therefore, we conceived a system-wide background daemon (`fluidsync-daemon`). By pre-aggregating system load and providing dynamic layout budgets via nanosecond Shared Memory reads, `dlangui` can seamlessly scale its UI complexity perfectly in real-time.

== 1. Disabling Internal OS Redraw Smoothing (The "Sizing Trick")

**Branch:** `feature/fluidsync-integration`

*Goal:* Wrest control of the resizing operation away from the OS to eliminate backbuffer stretching.

=== Changes
* Add handlers for `WM_SIZING`, `WM_ENTERSIZEMOVE`, and `WM_EXITSIZEMOVE` inside the `win32window.d` messaging loop wrapper.
* When `WM_SIZING` is intercepted, call the renderer's `invalidate()` and invoke a synchronous internal redraw call explicitly before passing the handle back to `DefWindowProc`. 
* Ensure class styles `CS_VREDRAW` and `CS_HREDRAW` are effectively managed (and evaluate intercepting `WM_ERASEBKGND` to return a `HBRUSH` matching the theme to prevent flashing).

== 2. Dynamic Budgeting (The "Skeleton" Phase)

**Branch:** `feature/dynamic-budgeting`

*Goal:* Prevent the synchronous `WM_SIZING` calls from locking the thread by introducing a fast, low-fidelity rendering pass driven by `fluidsync-daemon` telemetry.

=== Changes
* Introduce an `IUIBudgetProvider` interface connected directly to the `fluidsync` mapped memory block (with local socket fallback).
* Alter the internal `Widget.layout()` and `Widget.onDraw()` mechanisms. Add a parameter (e.g., `bool isSkeletonPass`) or check a global framework state (`Application.getInstance().isUrgentResize()`).
* **Heuristic Rules During Urgent Resizes:**
  - Standard structural layouts (Margins, VerticalLayouts, Splitters, Backgrounds) proceed normally.
  - Deep tree calculations (Font bounds calculations inside large arrays, heavy child iteration in TableLayout) are instantly aggressively skipped or debounced. 
  - Offscreen caching logic is updated so a widget returns its "last known" texture bounding box instead of recalculating its internals during the skeleton phase.

== 3. Multithreaded Rasterization (The Grand Decoupling)

**Branch:** `feature/multithreaded-renderer`

*Goal:* A single-threaded DlangUI application will always eventually drop frames if layout (CPU calculation) and rasterization (putting pixels in a buffer) occur sequentially on the same thread. We must establish an off-main-thread rasterizer for ultimate fluidity.

=== Changes
* **Thread Roles:** The Main UI Event Thread is restricted strictly to managing the logical Widget Tree, layout bounds, and input event polling.
* **Render Thread:** A distinct background thread is spawned per Window to oversee the OpenGL contexts (or Software rasterization surfaces). 
* **The Render Queue:** When the Main Thread finishes a layout calculation pass, it freezes and publishes a snapshot of the *Display List* (a flat array of fast primitive commands: `drawRect`, `drawTextLabel`, `clipBounds`).
* The Render Thread asynchronously consumes this Display List and rasterizes it to the backbuffer. This allows the Main Thread to immediately loop back and process the next `WM_MOUSEMOVE` or `WM_SIZING` event cleanly without waiting for pixel math to finish, effectively resolving the stutter forever.

== Prioritization Strategy
These features must be iterated incrementally to prevent regressions:
1. Lock the frame via OS tricks to expose the stutter.
2. Integrate `fluidsync` dynamic budgeting to mitigate the stutter intelligently.
3. Establish multithreading for permanent architectural fluidity.
